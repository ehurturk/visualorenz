					<div id="mw-content-text" class="mw-body-content"><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Normalizing a database design to reduce the duplication of data and ensure referential integrity</div>
<p><b>Third normal form</b> (<b>3NF</b>) is a <a href="/wiki/Database_schema" title="Database schema">database schema</a> design approach for <a href="/wiki/Relational_database" title="Relational database">relational databases</a> which uses <a href="/wiki/Database_normalization" title="Database normalization">normalizing</a> principles to reduce the duplication of data, avoid <a href="/wiki/Data_anomaly" class="mw-redirect" title="Data anomaly">data anomalies</a>, ensure <a href="/wiki/Referential_integrity" title="Referential integrity">referential integrity</a>, and simplify <a href="/wiki/Data_management" title="Data management">data management</a>. It was defined in 1971 by <a href="/wiki/Edgar_F._Codd" title="Edgar F. Codd">Edgar F. Codd</a>, an English computer scientist who invented the <a href="/wiki/Relational_model" title="Relational model">relational model</a> for <a href="/wiki/Database" title="Database">database</a> management.
</p><p>A <a href="/wiki/Relation_(database)" title="Relation (database)">database relation</a> (e.g. a <a href="/wiki/Table_(database)" title="Table (database)">database table</a>) is said to meet third normal form standards if all the attributes (e.g. <a href="/wiki/Column_(database)" title="Column (database)">database columns</a>) are <a href="/wiki/Functional_dependency" title="Functional dependency">functionally dependent</a> on solely a <a href="/wiki/Candidate_key" title="Candidate key">key</a>, except the case of functional dependency whose right hand side is a prime attribute (an attribute which is strictly included into some key). Codd defined this as a relation in <a href="/wiki/Second_normal_form" title="Second normal form">second normal form</a> where all non-prime attributes depend only on the <a href="/wiki/Candidate_key" title="Candidate key">candidate keys</a> and do not have a <a href="/wiki/Transitive_dependency" title="Transitive dependency">transitive dependency</a> on another key.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> 
</p><p>A hypothetical example of a failure to meet third normal form would be a hospital database having a table of patients which included a column for the telephone number of their doctor. (The phone number is dependent on the doctor, rather than the patient, thus would be better stored in a table of doctors.) The negative outcome of such a design is that a doctor's number will be duplicated in the database if they have multiple patients, thus increasing both the chance of input error and the cost and risk of updating that number should it change (compared to a third normal form-compliant data model that only stores a doctor's number once on a doctor table).
</p><p>Codd later realized that 3NF did not eliminate all undesirable data anomalies and developed a stronger version to address this in 1974, known as <a href="/wiki/Boyce%E2%80%93Codd_normal_form" title="Boyce–Codd normal form">Boyce–Codd normal form</a>.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Definition_of_third_normal_form">Definition of third normal form</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Third_normal_form&amp;action=edit&amp;section=1" title="Edit section: Definition of third normal form"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The third normal form (3NF) is a <a href="/wiki/Database_normalization#Normal_forms" title="Database normalization">normal form</a> used in <a href="/wiki/Database_normalization" title="Database normalization">database normalization</a>. 3NF was originally defined by <a href="/wiki/E._F._Codd" class="mw-redirect" title="E. F. Codd">E.&#160;F. Codd</a> in 1971.<sup id="cite_ref-Codd_2-0" class="reference"><a href="#cite_note-Codd-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> 
</p><p>Codd's definition states that a table is in 3NF <a href="/wiki/If_and_only_if" title="If and only if">if and only if</a> both of the following conditions hold:
</p>
<ul><li>The <a href="/wiki/Relation_(database)" title="Relation (database)">relation</a> R (table) is in <a href="/wiki/Second_normal_form" title="Second normal form">second normal form</a> (2NF).</li>
<li>No non-prime attribute of R is transitively dependent on the primary key.</li></ul>
<p>A <i>non-prime attribute</i> of R is an attribute that does not belong to any <a href="/wiki/Candidate_key" title="Candidate key">candidate key</a> of R.<sup id="cite_ref-Codd2_3-0" class="reference"><a href="#cite_note-Codd2-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> A <a href="/wiki/Transitive_dependency" title="Transitive dependency">transitive dependency</a> is a <a href="/wiki/Functional_dependency" title="Functional dependency">functional dependency</a> in which <i>X</i> → <i>Z</i> (<i>X</i> determines <i>Z</i>) indirectly, by virtue of <i>X</i> → <i>Y</i> and <i>Y</i> → <i>Z</i> (where it is not the case that <i>Y</i> → <i>X</i>).<sup id="cite_ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>
</p><p>A 3NF definition that is equivalent to Codd's, but expressed differently, was given by Carlo Zaniolo in 1982. This definition states that a table is in 3NF if and only if for each of its functional dependencies <i>X</i> → <i>Y</i>, at least one of the following conditions holds:<sup id="cite_ref-Zaniolo_5-0" class="reference"><a href="#cite_note-Zaniolo-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability"><span title="Quotation needed from source to verify. (November 2010)">need quotation to verify</span></a></i>&#93;</sup>
</p>
<ul><li><i>X</i> contains <i>Y</i> (that is, <i>Y</i> is a subset of <i>X</i>, meaning <i>X</i> → <i>Y</i> is a trivial functional dependency),</li>
<li><i>X</i> is a <a href="/wiki/Superkey" title="Superkey">superkey</a>,</li>
<li>every element of <i>Y</i>&#160;\&#160;<i>X</i>, the <a href="/wiki/Complement_(set_theory)#Relative_complement" title="Complement (set theory)">set difference</a> between Y and X, is a <i>prime attribute</i> (i.e., each attribute in <i>Y</i>&#160;\&#160;<i>X</i> is contained in some <a href="/wiki/Candidate_key" title="Candidate key">candidate key</a>).</li></ul>
<p>To rephrase Zaniolo's definition more simply, the relation is in 3NF if and only if for every non-trivial functional dependency X → Y, X is a superkey or <i>Y</i>&#160;\&#160;<i>X</i> consists of prime attributes. Zaniolo's definition gives a clear sense of the difference between 3NF and the more stringent <a href="/wiki/Boyce%E2%80%93Codd_normal_form" title="Boyce–Codd normal form">Boyce–Codd normal form</a> (BCNF). BCNF simply eliminates the third alternative ("Every element of <i>Y</i>&#160;\&#160;<i>X</i>, the set difference between <i>Y</i> and <i>X</i>, is a prime attribute.").
</p>
<div class="mw-heading mw-heading2"><h2 id="&quot;Nothing_but_the_key&quot;"><span id=".22Nothing_but_the_key.22"></span>"Nothing but the key"</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Third_normal_form&amp;action=edit&amp;section=2" title="Edit section: &quot;Nothing but the key&quot;"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>An approximation of Codd's definition of 3NF, paralleling the traditional <a href="/wiki/Sworn_testimony" title="Sworn testimony">oath</a> to give true evidence in a court of law, was given by Bill Kent: "[every] non-key [attribute] must provide a fact about the key, the whole key, and nothing but the key".<sup id="cite_ref-Kent_7-0" class="reference"><a href="#cite_note-Kent-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> A common variation supplements this definition with the oath "so help me <a href="/wiki/Edgar_F._Codd" title="Edgar F. Codd">Codd</a>".<sup id="cite_ref-Diehr_8-0" class="reference"><a href="#cite_note-Diehr-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Requiring existence of "the key" and requiring that non-key attributes be dependent on "the whole key" ensures <a href="/wiki/Second_normal_form" title="Second normal form">2NF</a>; further requiring that non-key attributes be dependent on "nothing but the key" ensures 3NF. While this phrase is a useful mnemonic, the fact that it only mentions a single key means it defines some <a href="/wiki/Necessity_and_sufficiency" title="Necessity and sufficiency">necessary but not sufficient conditions</a> to satisfy the 2nd and 3rd normal forms. Both 2NF and 3NF are concerned equally with <i>all</i> candidate keys of a table and not just any one key.
</p><p><a href="/wiki/Christopher_J._Date" title="Christopher J. Date">Chris Date</a> refers to Kent's summary as "an intuitively attractive characterization" of 3NF and notes that with slight adaptation it may serve as a definition of the slightly stronger <a href="/wiki/Boyce%E2%80%93Codd_normal_form" title="Boyce–Codd normal form">Boyce–Codd normal form</a>: "Each attribute must represent a fact about the key, the whole key, and nothing but the key."<sup id="cite_ref-DateIntro_9-0" class="reference"><a href="#cite_note-DateIntro-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> The 3NF version of the definition is weaker than Date's BCNF variation, as the former is concerned only with ensuring that <i>non-key</i> attributes are dependent on keys. Prime attributes (which are keys or parts of keys) must not be functionally dependent at all; they each represent a fact about the key in the sense of providing part or all of the key itself. (This rule applies only to functionally dependent attributes, as applying it to all attributes would implicitly prohibit composite candidate keys, since each part of any such key would violate the "whole key" clause.)

<div class="mw-heading mw-heading2"><h2 id="Computation">Computation</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Third_normal_form&amp;action=edit&amp;section=3" title="Edit section: Computation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>A relation can always be decomposed in third normal form, that is, the relation R is rewritten to <a href="/wiki/Relational_projection" class="mw-redirect" title="Relational projection">projections</a> R<sub>1</sub>, ..., R<sub>n</sub> whose <a href="/wiki/Natural_join" class="mw-redirect" title="Natural join">join</a> is equal to the original relation. Further, this decomposition does not lose any <a href="/wiki/Functional_dependencies" class="mw-redirect" title="Functional dependencies">functional dependency</a>, in the sense that every functional dependency on R can be derived from the functional dependencies that hold on the projections R<sub>1</sub>, ..., R<sub>n</sub>. What is more, such a decomposition can be computed in <a href="/wiki/Polynomial_time" class="mw-redirect" title="Polynomial time">polynomial time</a>.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>
</p><p>To decompose a relation into 3NF from 2NF, break the table into the <a href="/wiki/Canonical_cover" title="Canonical cover">canonical cover</a> functional dependencies, then create a relation for every candidate key of the original relation which was not already a subset of a relation in the decomposition.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Equivalence_of_the_Codd_and_Zaniolo_definitions_of_3NF">Equivalence of the Codd and Zaniolo definitions of 3NF</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Third_normal_form&amp;action=edit&amp;section=4" title="Edit section: Equivalence of the Codd and Zaniolo definitions of 3NF"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The definition of 3NF offered by Carlo Zaniolo in 1982, and given above, can be shown to be equivalent to the Codd definition in the following way: Let X → A be a nontrivial <a href="/wiki/Functional_dependency" title="Functional dependency">FD</a> (i.e. one where X does not contain A) and let A be a non-prime attribute. Also let Y be a candidate key of R. Then Y → X. Therefore, A is not transitively dependent on Y if there is a functional dependency X → Y iff X is a superkey of R.
</p>
<div class="mw-heading mw-heading2"><h2 id="Normalization_beyond_3NF">Normalization beyond 3NF</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Third_normal_form&amp;action=edit&amp;section=5" title="Edit section: Normalization beyond 3NF"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Most 3NF tables are free of update, insertion, and deletion anomalies. Certain types of 3NF tables, rarely met with in practice, are affected by such anomalies; these are tables which either fall short of <a href="/wiki/Boyce%E2%80%93Codd_normal_form" title="Boyce–Codd normal form">Boyce–Codd normal form</a> (BCNF) or, if they meet BCNF, fall short of the higher normal forms <a href="/wiki/Fourth_normal_form" title="Fourth normal form">4NF</a> or <a href="/wiki/Fifth_normal_form" title="Fifth normal form">5NF</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="Considerations_for_use_in_reporting_environments">Considerations for use in reporting environments</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Third_normal_form&amp;action=edit&amp;section=6" title="Edit section: Considerations for use in reporting environments"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>While 3NF was ideal for machine processing, the segmented nature of the data model can be difficult to intuitively consume by a human user. <a href="/wiki/Analytics" title="Analytics">Analytics</a> via query, reporting, and dashboards were often facilitated by a different type of data model that provided pre-calculated analysis such as trend lines, period-to-date calculations (month-to-date, quarter-to-date, year-to-date), cumulative calculations, basic statistics (average, standard deviation, moving averages) and previous period comparisons (year ago, month ago, week ago) e.g. <a href="/wiki/Dimensional_modeling" title="Dimensional modeling">dimensional modeling</a> and beyond dimensional modeling, flattening of stars via <a href="/wiki/Hadoop" class="mw-redirect" title="Hadoop">Hadoop</a> and <a href="/wiki/Data_science" title="Data science">data science</a>.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup><sup id="cite_ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> Hadley Wickham's "tidy data" framework is 3NF, with "the constraints framed in statistical language".<sup id="cite_ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup>
</p>
